Python版を基準とした各言語の実装比較

## Python -> Numba での変更点

Numba版はPure Python版に対して最小限の変更で高速化を実現している。
以下、4つのベンチマークそれぞれの変更点を示す。

### モンテカルロ法

変更点は2箇所のみ。

- `from numba import njit` のインポート追加
- 関数に `@njit` デコレータを付与
- ウォームアップ呼び出し `monte_carlo_pi(1000)` を追加

関数本体のコードは一切変更なし。`random.random()` もNumba内でそのまま動作する。

### シンプソン法

モンテカルロ法と同じパターン。

- `from numba import njit` のインポート追加
- 関数に `@njit` デコレータを付与
- ウォームアップ呼び出し `simpson(1000)` を追加

関数本体のコードは一切変更なし。

### N体問題

他のベンチマークと異なり、データ構造の変更が必要になる。

- **データ構造の変更**: Pure Python版はリストのリスト(`list[list[float]]`)で各天体を表現するが、Numba版では `numpy.ndarray` (5x7の2次元配列)に変更。Numbaは任意のPythonオブジェクトを扱えないため、NumPy配列への変換が必要。
- **関数の分離**: Pure Python版ではトップレベルに直接ループを書いているが、Numba版ではメインループを `run()` 関数に切り出して `@njit` を付与。
- **3関数に `@njit`**: `advance`, `energy`, `run` の3関数それぞれにデコレータを付与。
- **ウォームアップ**: 3関数すべてを事前に小さい引数で呼び出し、JITコンパイルを完了させる。
- **アクセス方法の変更**: `body[0]` (リストインデックス) から `bodies[i, 0]` (NumPy 2次元インデックス) への書き換え。

このベンチマークではNumba化に際して、単なるデコレータ付与ではなくデータ構造の再設計が求められる。Numbaが効率的に扱える型（NumPyのプリミティブ配列）に合わせる作業が発生する。

### Rule 110

Rule 110では、Numba最小版と最適化版の2段階を用意した。

**Numba最小版** (`rule110_numba_minimal.py`):

- `list[int]` を `numpy.ndarray` (`dtype=np.int32`) に変更（Numbaがリストを効率的に扱えないため最低限必要）
- ルールテーブルもNumPy配列に変更
- 関数全体を1つの `@njit` 関数にまとめる（分離なし）
- ウォームアップ呼び出しを追加

ロジック自体はPure Python版とほぼ同一。`@njit` を付けただけに近い最小変更。

**Numba最適化版** (`rule110_numba.py`):

- 1世代の更新を `step()` 関数に、世代ループ全体を `run()` 関数に分離
- ルールテーブルを関数の外でNumPy配列として定義し、引数として渡す
- ウォームアップは小さいサイズ (幅100, 10世代) で事前呼び出し

ベンチマーク結果では、最小版と最適化版の性能差はほぼなかった（376.8 ms vs 363.9 ms）。この題材では関数分離の最適化効果は限定的で、`@njit` デコレータ付与が性能向上の主因であることが分かる。

### Numba化のまとめ

| 変更の種類 | モンテカルロ | シンプソン | N体 | Rule 110 最小 | Rule 110 最適化 |
|---|---|---|---|---|---|
| `@njit` デコレータ追加 | o | o | o | o | o |
| ウォームアップ呼び出し | o | o | o | o | o |
| データ構造の変更 | - | - | o | o | o |
| 関数の分離・再設計 | - | - | o | - | o |
| NumPyインポート追加 | - | - | o | o | o |

スカラー値のみを扱うベンチマーク(モンテカルロ、シンプソン)ではデコレータを付けるだけで済む。コレクション型(リスト、配列)を扱うベンチマーク(N体、Rule 110)ではNumPy配列への変換が最低限必要だが、関数分離まで踏み込んだ最適化の効果は題材に依存する。

## Python -> NumPy での変更点

NumPy版はループをベクトル演算に置き換えることで高速化を図る。ただし効果は問題構造に大きく依存する。

### モンテカルロ法 -- ベクトル化が有効なケース

ループを完全に排除し、全計算をベクトル演算に置き換え。

- `random.random()` のループ -> `np.random.random(n)` で一括生成
- `if x*x + y*y <= 1.0` の条件判定 -> `np.sum(x*x + y*y <= 1.0)` でベクトル判定
- ループゼロの完全ベクトル化

結果: **137.1 ms**（C++の3.7倍）。Pure Pythonの565 msから4.1倍高速化。Numba（297.8 ms）よりも速い。

### シンプソン法 -- ベクトル化が有効なケース

同様にループを完全に排除。

- `for i in range(1, n)` のループ -> `np.arange(1, n) * h` で全点を一括生成
- 条件分岐による係数計算 -> `np.where(indices % 2 == 0, 2.0, 4.0)` でベクトル化
- `np.sum(coeffs * f(x))` で一括積算

結果: **559.4 ms**（C++の18.1倍）。Pure Pythonの2,277 msから4.1倍高速化。ただし大量のメモリ確保（5000万要素の配列）がオーバーヘッドとなり、Numba（274.2 ms）には及ばない。

### Rule 110 -- ベクトル化が有効なケース

世代ループは残るが、各世代内のセル更新をベクトル化。

- セルごとのループ -> `np.roll(current, 1)`, `np.roll(current, -1)` でシフト配列を一括生成
- `left * 4 + center * 2 + right` をベクトル演算でインデックス計算
- `rule[index]` のNumPy fancy indexingでルール適用

結果: **314.1 ms**（C++の8.1倍）。Pure Pythonの5,747 msから18.3倍高速化。Numba（363.9 ms）よりも速い。セルオートマトンは配列全体に対する一括演算と相性が良い。

### N体問題 -- ベクトル化が非効果的なケース

5体間の相互作用を計算するが、速度更新は逐次的に行う必要がある。

- 天体データをNumPy配列で表現（Pure Pythonのリストのリストから変更）
- ペアワイズ距離計算にNumPy演算を使用
- ただし力の更新ループは逐次依存があり排除できない

結果: **17,483 ms**（Rustの556倍）。Pure Pythonの3,026 msよりも**大幅に遅い**。5体しかないためベクトル化のメリットがなく、NumPy関数呼び出しのオーバーヘッドだけが加算される。

### NumPyベクトル化のまとめ

| ベンチマーク | Pure Python | NumPy | 高速化率 | Numbaとの比較 |
|---|---|---|---|---|
| モンテカルロ | 565 ms | 137 ms | 4.1倍 | NumPyの方が速い |
| シンプソン | 2,277 ms | 559 ms | 4.1倍 | Numbaの方が速い |
| Rule 110 | 5,747 ms | 314 ms | 18.3倍 | NumPyの方が速い |
| N体 | 3,026 ms | 17,483 ms | **0.17倍（悪化）** | Numbaの方が圧倒的に速い |

NumPyベクトル化は、内側ループを配列演算に完全に置き換えられる問題では効果が高い。一方、逐次依存のある計算や要素数が少ない場合は関数呼び出しオーバーヘッドが支配的になり、逆効果になる。

## Julia ベクトル化版での変更点

Juliaのブロードキャスト（`.` 演算子）版を、モンテカルロ法とシンプソン法で作成した。

### モンテカルロ法

- `rand()` のループ -> `rand(n)` で一括生成
- `sum(@. x*x + y*y <= 1.0)` でベクトル判定
- `@.` マクロは式全体をブロードキャストに展開する

結果: **287.6 ms**（ループ版123.5 msの2.3倍遅い）。

### シンプソン法

- `collect(1:(n-1)) .* h` でベクトル生成
- `ifelse.(isodd.(1:(n-1)), 4.0, 2.0)` で係数ベクトル
- `sum(coeffs .* 4.0 ./ (1.0 .+ x .* x))` で一括計算

結果: **691.6 ms**（ループ版138.0 msの5.0倍遅い）。

### Juliaではベクトル化が遅い理由

Juliaのループは既にJITで最適化されており、C++/Rustに近い性能が出る。ベクトル化版は大量の一時配列を確保するため、メモリアロケーションとGCのオーバーヘッドが加わる。PythonではNumPyのC実装がインタプリタのオーバーヘッドを回避するためベクトル化が有効だが、Juliaではその必要がない。

## C++/Rust のベクトル化について

C++/Rustにはベクトル化版を追加していない。理由:

- コンパイラの `-O3` / `opt-level=3` でSIMDオートベクトル化が効いている
- 「NumPyベクトル化」はPythonインタプリタのオーバーヘッドを回避する手法であり、ネイティブコンパイル言語では対応する概念がない
- Eigen (C++) や ndarray (Rust) を使う版は可能だが、ライブラリの性能比較になり、同一アルゴリズムの言語間比較という本ベンチマークの趣旨からずれる

## Julia / C++ / Rust で記述量が増える箇所

### Julia

Juliaのコードは全体的にPythonと記述量がほぼ同等で、増加する箇所は少ない。

- **型注釈**: 関数引数に `n::Int` のように型を明示する。Pythonのtype hintと同程度だが、Juliaでは性能に直結するため省略しにくい。
- **1-based indexing**: 配列のインデックスが1始まりのため、Rule 110のルールテーブル参照で `rule[index + 1]` とオフセットが必要。
- **`@inbounds` マクロ**: ループ内の配列アクセスで境界チェックを省略するマクロ。パフォーマンスのために付与するが、Pythonにはない記述。N体問題とRule 110で使用。
- **`mutable struct`**: N体問題では`mutable struct Body` を定義する。Pythonのリストと異なり、構造体の定義が明示的に必要。

一方、Juliaでは以下が簡潔になる。

- `for _ in 1:n` は `for _ in range(n)` とほぼ同等
- 三項演算子 `i > 1 ? current[i-1] : 0` はPythonより短い
- `println` と f-string相当の `"$var"` 構文は同程度

### C++

C++はPythonと比較して記述量が明確に増える。

- **型宣言**: すべての変数に型が必要。`double x = dist(rng);` のように、Pythonの `x = random.random()` に比べて冗長。
- **struct定義**: N体問題で `struct Body` のメンバを明示的に列挙。7つのフィールド (`x, y, z, vx, vy, vz, mass`) をすべて型付きで宣言する必要がある。
- **ヘッダインクルード**: `#include <cstdio>`, `#include <cmath>`, `#include <random>`, `#include <vector>` 等。Pythonの `import` より細かい粒度でインクルードが必要。
- **main関数**: `int main() { ... return 0; }` のボイラープレート。
- **乱数生成**: Pythonの `random.random()` 1行に対し、C++では `std::mt19937_64 rng(42);` と `std::uniform_real_distribution<double> dist(0.0, 1.0);` の2行の初期化が必要。
- **定数宣言**: `constexpr double PI = 3.141592653589793;` のように、`constexpr` と型を明記。Pythonの `SOLAR_MASS = 4 * math.pi * math.pi` より冗長。
- **出力書式**: `std::printf("Pi ≈ %.10f\n", pi_est);` は `print(f"Pi ≈ {pi_est}")` に比べて書式指定子が必要。

### Rust

RustはC++と同程度の記述量だが、以下の特徴がある。

- **所有権・ミュータビリティ**: `let mut bodies` のように変更可能な変数には `mut` を明示。関数引数でも `bodies: &mut [Body; N]` と参照の種類を指定。
- **型変換**: `i as f64`, `count as f64` のように整数と浮動小数点の変換が明示的。Pythonでは暗黙的に行われる処理。
- **derive属性**: `#[derive(Clone, Copy)]` を構造体に付与する必要がある(N体問題)。
- **イテレータ記法**: `current.iter().map(|&x| x as u64).sum()` のようなイテレータチェーン。Pythonの `sum(current)` と比べて記述量が増える。

### 言語間の記述量比較

N体問題を例にとると:

| 言語 | 行数(概算) | 増加要因 |
|---|---|---|
| Python | 68行 | - (基準) |
| Python (Numba) | 80行 | ウォームアップ、関数分離 |
| Julia | 65行 | mutable struct定義があるが、全体的に簡潔 |
| C++ | 115行 | struct定義、型宣言、ヘッダ、main関数 |
| Rust | 137行 | struct定義、型変換、所有権、derive |

シンプソン法のような単純なループ処理では言語間の差は小さくなる:

| 言語 | 行数(概算) |
|---|---|
| Python | 15行 |
| Python (Numba) | 21行 |
| Julia | 17行 |
| C++ | 22行 |
| Rust | 19行 |
